As-Is / To-Be flows

Two core processes mapped in their current and proposed states. Value is expressed in concrete numbers: steps saved, channels merged, and failure points eliminated. (BRD §7)

Process 1 · Shift schedule change

A dispatcher changes a driver's shift — how it reaches them, and what happens along the way

As-Is — How it works today
01Dispatcher updates the schedule in the internal system.⚠ No automatic trigger to the driver
02Dispatcher emails the driver the updated PDF.⚠ Lands in personal inbox, not guaranteed to be read
03If urgent — also posts in the WhatsApp group.⚠ Lost in group noise
04Sometimes also SMS to personal phone.⚠ No read-receipt
05Dispatcher calls to confirm the message landed.⚠ Manual call, unknown date and time
06If the driver didn't answer — tries additional channels.⚠ No guaranteed cadence, no SLA
To-Be — With the Portal
01Dispatcher updates the schedule in the internal system.✓ Automatic API trigger to the Portal
02Push notification sent to the driver's app within 5 seconds.✓ Unified, logged channel
03Driver opens the app — read-receipt fires automatically and the dispatcher sees confirmation on the dashboard in real time.✓ Immediate visibility, zero calls
04If unread 5 hours before shift start — automatic push retry to the driver.✓ Second attempt built into the workflow
05If the second attempt also fails — automatic escalation to supervisor and unit representative.✓ Time-based SLA from shift, not minutes
Gap

6 steps → 5. 4 phone calls and two parallel channels → zero. Dispatcher visibility moved from "I hope" to "I know." This isn't just time savings — it's a reduction in operational risk.

Process 2 · Coordinating a taxi to shift

A driver needs a ride to their starting station — how it happens today, and how it should happen

As-Is — How it works today
01Driver calls the ops center to request a taxi.⚠ ~3 min call including hold time
02Ops agent logs the request in a journal/Excel.⚠ Manual, no consistent format
03Ops agent calls the taxi company.⚠ Another 2 min minimum
04Taxi company assigns a vehicle, notifies ops.⚠ Assignment time unknown upfront
05Ops calls the driver back to update.⚠ Fourth call on the same request
06Driver waits, calls again if taxi is late.⚠ Repeat calls, stress
To-Be — With the Portal
01Driver opens the app, taps "Request a ride" from today's screen.✓ Data (date, station, time) pre-filled automatically
02The system sends the request directly to the taxi company's API.✓ Zero human involvement in ops
03Within 30 seconds, the driver receives confirmation with vehicle number and ETA.✓ Immediate visibility
04Taxi status updates in real time on screen.✓ No repeat calls
05Exceptions only (urgent request / issue) — escalation to ops.✓ Ops handles only what requires judgment
Gap

6 steps → 5. 4 phone calls → zero in the normal flow. ~60 daily calls to the ops center eliminated. This isn't just operational savings — it's returning the ops center to serving exceptions, which is what it was meant to be in the first place.

A pattern shared by both processes

In both processes, an organizational worker functions as a manual interface between source systems that already exist in the organization. A dispatcher translates a change in one system into three communication platforms; an ops agent translates a driver's call into a call to the scheduling system. These translation steps are the capacity this project frees up.

The platform moves repetitive activity onto systemic infrastructure. Human capital in ops stays available for what requires judgment — exceptions, decisions, edge cases.
