Artifact 04 · Integration Specification

חמש אינטגרציות עם מערכות מקור

ה-Portal לא מחזיק נתונים משלו — הוא חלון לכמה מערכות שכבר קיימות בארגון. לכל אינטגרציה הוגדרו: סוג חיבור, כיוון נתונים, תדירות, ומסלול fallback אם המערכת המקור לא זמינה. (BRD §5)

חמש האינטגרציות, אחת אחת

INT-01
Scheduling System
מערכת סידור עבודה — מקור הסידור היומי

שולחת ל-Portal את סידור העבודה בכל פעם שיש שינוי. זה ה-core data flow שמזין את כל מסכי ה"היום ומחר" של הנהג.

סוג
REST API (JSON) — push on change
תדירות
אירועי-דחיפה (בכל שינוי) + sync יומי לבדיקת עקביות
זרימה
Scheduling → Portal: שיוך נהג למשמרת, שינויים, הוספות, ביטולים
אימות
shift_id ייחודי + checksum על כל ה-payload
Fallback: אם ה-API לא זמין — ייבוא ידני של Excel ע"י דיספצ'ר, פעם ביום. נשמרת אותה סכמת רשומות.
INT-02
Dispatch Operations
מוקד תפעולי — תיאום הסעות (מוניות / רכבי איסוף)

שני כיוונים: נהג שולח בקשת איסוף דרך ה-Portal, מוקד מאשר/מקצה רכב, ועדכון סטטוס חוזר ל-Portal עד הגעת הרכב. ETA למונית בלילה לא יכול לסבול 15 דקות עיכוב — לכן הארכיטקטורה כאן event-driven מהיסוד.

סוג
Webhook (ראשי) + REST polling כ-fallback בלבד
תדירות
Webhook על כל שינוי סטטוס (real-time). Polling פעם ב-10 דק' רק לאיתור אירועים שאבדו (network blip, downtime).
זרימה
Portal → Dispatch: בקשה. Dispatch → Portal (webhook): אישור, רכב, ETA, סטטוס חי בכל שינוי.
אימות
OAuth2 + driver_id משויך לבקשה; webhook חתום עם HMAC לאימות מקור
Fallback: אם ה-webhook נופל לאורך זמן (>15 דק' ללא פעימה) — מעבר אוטומטי ל-polling אגרסיבי (פעם בדקה) + פאנל admin ידני במוקד לעדכון סטטוס.
INT-03
HR System of Record
מערכת משאבי אנוש — SSO + REST

מקור-האמת לכל מה שקשור לנהג כעובד: יתרת חופשה, תלושי שכר, חגים, זכאויות. ה-Portal לא מחזיק עותק — הוא קורא וכותב בקשות חופשה ישירות לכאן.

סוג
SSO ל-login + REST API ל-data
תדירות
on-demand (כל לחיצה של הנהג על "יתרת חופשה")
זרימה
HR → Portal: יתרה, תלושים. Portal → HR: בקשת חופשה חדשה.
אימות
SSO token; refresh כל 30 דק'
Fallback: ייצוא CSV שבועי לקריאה בלבד; פעולות (הגשת בקשה) מושבתות עד שה-API חוזר.
INT-04
FCM / APNs
שירותי Push Notifications של Google ו-Apple

שליחת התראות לנייד של הנהג: שינוי בסידור, אישור בקשה, הודעה תפעולית קריטית. כל התראה דורשת read-receipt חוזר ל-Portal.

סוג
Firebase Cloud Messaging (Android) + APNs (iOS)
תדירות
אירוע (push at event time)
זרימה
Portal → FCM/APNs → device. read-receipt חוזר דרך ה-App.
אימות
FCM server key + APNs cert; rotation תקופתית
Fallback: הודעה מוצגת ב-In-App Inbox בכניסה הבאה לאפליקציה גם אם ה-push נכשל.
INT-05
Organizational IDP
ספק הזהויות הארגוני (Active Directory / Azure AD)

זיהוי ראשוני של המשתמש לפני שמירת session ב-Portal. אין סיסמה ייעודית ל-Portal — משתמשים בזיהוי הארגוני שכבר קיים לכל עובד. זה גם העיקרון שמכתיב את מנגנון ה-Fallback שלמטה.

סוג
SAML 2.0 או OIDC (תלוי בתשתית הקיימת)
תדירות
פעם ב-login; refresh לפי policy ארגוני
זרימה
Browser/App → IDP → Portal session
אימות
X.509 signing cert; metadata exchange חד-פעמי
Fallback (מובדל לפי תפקיד — כי IDP למטה לא מאפשר אימות זהות אמיתי):
• דיספצ'רים ומנהלים — Emergency Access עם MFA משני (טוקן חומרה / קוד OTP מראש) תחת בקרת מנהל אבטחה. מבטיח נראות תפעולית גם בזמן downtime.
• נהגים — לא מקבלים Emergency Access ל-Portal (בניית מערכת ניהול משתמשים מקבילה תסתור את עיקרון "Portal לא מחזיק נתונים"). במקום זה: סידור היומי נשלח אוטומטית כ-SMS, ומוקד תפעולי מקבל את כל הפניות בטלפון עד שה-IDP חוזר.

טבלת ריכוז — חמש האינטגרציות במבט אחד

סיכום של כל המפרטים שלמעלה. שימושי כדף עזר לדיון Steering Committee, ולתיעוד כניסה לפיתוח.

מזהה מערכת מקור סוג חיבור כיוון Fallback
INT-01 Scheduling REST (Push) מערכת ← פורטל העלאת Excel ידנית ע"י דיספצ'ר
INT-02 Dispatch Webhook (ראשי) + REST דו-כיווני Polling אגרסיבי + פאנל ידני במוקד
INT-03 HR System SSO + REST דו-כיווני קריאה בלבד (CSV), חסימת כתיבה
INT-04 FCM / APNs Push Notification פורטל ← מכשיר In-App Inbox (קריאה בכניסה הבאה)
INT-05 Org IDP SAML 2.0 / OIDC משתמש ← IDP Emergency Access לניהול (MFA); נהגים — SMS אוטומטי

למה ה-fallbacks חיו במפרט מהשורה הראשונה

לפני שאישרתי שורה אחת של פונקציונליות, שאלתי על כל אינטגרציה אותה שאלה: מה קורה כשהצד השני לא זמין? התשובה לא יכולה להיות "מערכת לא זמינה — נא לפנות לתמיכה". זה מסיט את הסיכון התפעולי לנהג שבדיוק עומד להתחיל משמרת.

כל fallback בטבלאות למעלה הוא תוצר של דרישה אופרטיבית אחת: המערכת חייבת להמשיך לעבוד גם כשאחד הצדדים נופל. ייבוא Excel ידני, polling במקום webhook, ייצוא CSV שבועי, In-App Inbox במקום push — אלה לא פשרות. הם החלטות שמורידות את הסיכון לאפס בתסריטים שיקרו, לא אם, אלא מתי.

זה השוני בין מפרט שאישרתי לחתימה לבין מפרט שלמדתי על קיומו אחרי ה-go-live. את ההפרש הזה הארגון לא רוצה לגלות בשטח.
