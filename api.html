Integration Specification

Five integrations with source systems

The Portal doesn't hold its own data — it's a window into several systems that already exist in the organization. Each integration defines: connection type, data direction, frequency, and a fallback path if the source system isn't available. (BRD §5)

The five integrations, one by one

INT-01
Scheduling System
Work scheduling system — source of the daily schedule

Sends the work schedule to the Portal on every change. This is the core data flow that feeds all the driver's "today and tomorrow" screens.

Type
REST API (JSON) — push on change
Frequency
Push events (on every change) + daily sync for consistency check
Flow
Scheduling → Portal: driver-shift assignments, changes, additions, cancellations
Auth
Unique shift_id + checksum on the whole payload
Fallback: If the API isn't available — manual Excel import by dispatcher, once a day. Same record schema is preserved.
INT-02
Dispatch Operations
Operations center — ride coordination (taxis / pickup vehicles)

Two directions: driver sends a pickup request via the Portal, ops approves/assigns a vehicle, and status updates flow back to the Portal until the vehicle arrives. A taxi ETA at night can't tolerate a 15-minute delay — so the architecture here is event-driven from the start.

Type
Webhook (primary) + REST polling as fallback only
Frequency
Webhook on every status change (real-time). Polling every 10 min only to catch missed events (network blip, downtime).
Flow
Portal → Dispatch: request. Dispatch → Portal (webhook): approval, vehicle, ETA, live status on every change.
Auth
OAuth2 + driver_id bound to request; webhook signed with HMAC for source verification
Fallback: If the webhook fails over time (>15 min without a heartbeat) — automatic switch to aggressive polling (once per minute) + manual admin panel in ops for status updates.
INT-03
HR System of Record
HR system — SSO + REST

The source-of-truth for everything driver-as-employee: leave balance, payslips, holidays, entitlements. The Portal doesn't hold a copy — it reads from and writes leave requests directly here.

Type
SSO for login + REST API for data
Frequency
On-demand (every driver click on "leave balance")
Flow
HR → Portal: balance, payslips. Portal → HR: new leave request.
Auth
SSO token; refresh every 30 min
Fallback: Weekly CSV export for read-only access; actions (submitting requests) disabled until the API returns.
INT-04
FCM / APNs
Google and Apple Push Notification services

Sending notifications to the driver's phone: schedule changes, request approvals, critical operational messages. Every notification requires a read-receipt back to the Portal.

Type
Firebase Cloud Messaging (Android) + APNs (iOS)
Frequency
Event-based (push at event time)
Flow
Portal → FCM/APNs → device. Read-receipt comes back via the App.
Auth
FCM server key + APNs cert; periodic rotation
Fallback: Message displayed in the In-App Inbox on next app open, even if the push failed.
INT-05
Organizational IDP
Organizational identity provider (Active Directory / Azure AD)

Initial user authentication before storing a session in the Portal. No dedicated Portal password — uses the existing organizational identity that every employee already has. This is also the principle that dictates the fallback mechanism below.

Type
SAML 2.0 or OIDC (depends on existing infrastructure)
Frequency
Once per login; refresh per organizational policy
Flow
Browser/App → IDP → Portal session
Auth
X.509 signing cert; one-time metadata exchange
Fallback (differentiated by role — because IDP downtime doesn't allow real identity verification):
• Dispatchers and managers — Emergency Access with secondary MFA (hardware token / pre-shared OTP) under security manager control. Ensures operational visibility even during downtime.
• Drivers — Do not get Emergency Access to the Portal (building a parallel user management system would contradict the "Portal doesn't hold data" principle). Instead: daily schedule sent automatically as SMS, and the ops center handles all inquiries by phone until IDP returns.

Why fallbacks lived in the spec from day one

Before approving a single line of functionality, I asked the same question about every integration: what happens when the other side isn't available? The answer can't be "system unavailable — please contact support." That just transfers operational risk to the driver who's about to start their shift.

Every fallback in the tables above is the product of one operational requirement: the system must keep working even when one of the parties goes down. Manual Excel import, polling instead of webhook, weekly CSV export, In-App Inbox instead of push — these aren't compromises. They are decisions that reduce risk to zero in scenarios that will occur — not if, but when.

That's the difference between a spec I signed off on, and a spec I learned about after go-live. That gap is not one the organization wants to discover in the field.
